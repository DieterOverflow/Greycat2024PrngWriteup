Attempt 1 - Throwing statistical tests at a PRG

Wikipedia states:

[...] a pseudorandom generator (PRG) for a class of statistical tests is a deterministic procedure [...] such that no statistical test in the class can distinguish between the output of the generator and the uniform distribution. https://en.wikipedia.org/wiki/Pseudorandom_generator

So before understanding the challenge, we can throw a few known statistical tests at it. Maybe the challenge authors screwed up?

The NIST Statistical Testing Suite tests for True Random Number Generators (TRNGs) contain 15 statistical tests determining whether the randomness is good enough.

These range from a frequency test (test if the proportion of ones is 50) to a Discrete Fourier Transform (Spectral) Test* (???). Sadly, after throwing various GitHub repositories at it, the challenge PRG looks random enough - at least for short pseudo-random sequences.

In this section, we go on a small tangent and look at timing-based side-channel attacks. Feel free to skip this section. But I promise it is interesting!

Solution with smart matrix constraints

So I wanted to go to sleep but then the realization hit me.

Like with the z3 Attempt, the real solution creates a series of constraints and checks whether the random output could match these constraints. The difference here is that we leverage matrix multiplication and linear equations to create constraints that are solvable in a reasonable time.

After avoiding the task, let's finally look at the code: The code initializes x, r, and k with 8 random bytes. The code calculates the next bit by summing up the current x and multiplying a matrix A. The matrix A, defined in the file param.py, is a 64x64 matrix of 0s and 1s. The code adds r or k to x depending on the loop iteration.

As with all math problems, it helps to attempt to solve a more straightforward problem:

A simpler challenge

Assume the challenge would only generate x to calculate a random string like this:

def prg(length):
    x = token_bytes(8)
    x = np.array(bytes_to_bits(x))
    output = []
    for i in range(length * 8):
        output.append(sum(x) % 2)
        x = A @ x
    output = output
    return bits_to_bytes(output).hex()

Coding it in sagemath

A system of linear equations can be encoded as a matrix and solved via the solve_right function. The following code does that insagemath

In sagemath, given a matrix A, sum(A) returns the column sum vector colSum(A)⃗\vec{\text{{colSum}}(A)}colSum(A)​.

We can describe this set of Linear equations as a Matrix:

import param
from sage.all import *
I = matrix.identity(GF(2), 64)
A = Matrix(GF(2), param.A)
BITS = 16 * 8
x_matrix = []
current = I
for i in range(BITS):
    x_matrix.append(sum(current))
    current = A * current
x_matrix = Matrix(GF(2), x_matrix)

Given a bitstream rng of 128 bits, we can use solve_right to determine whether an x⃗\vec{x}x exists that solves x_matrix * x = rng. If a solution exists, rng is generated from the simple PRG; otherwise, rng is randomly generated.

The code to solve such a challenge would be:

The challenge case

In the simple case, we just looked at a problem where every bit was just created via x = A@x. The challenge adds additional complexity via r and s:

    for i in range(length * 8):
        output.append(sum(x) % 2)
        if (i % 3 == 0): x = (A @ x + r) % 2
        if (i % 3 == 1): x = (A @ x + k) % 2
        if (i % 3 == 2): x = (A @ x + r + k) % 2

Which results in these equations for the first few bits:
